# 설명 문서 — 항공기 정비 세션 아카이브 RAG

이 문서는 **무엇을 만들었고, 어떤 기능이며, 어떻게 동작하는지**를 정리한 것입니다.
코드를 읽기 전에 전체 그림을 잡는 용도로 보세요.

---

## 1. 한 줄 요약

> 정비사가 자연어로 질문하면, **과거 정비 사례와 매뉴얼을 의미 기반으로 검색**해서
> **근거(출처)와 함께** 답을 주고, 정비사가 검증한 결과를 다시 쌓는 **도메인 특화 RAG** 시스템.

핵심은 "AI가 많이 아는 것"이 아니라 **"정비사가 실제로 묻는 질문을 맥락과 함께 잘 축적·검색·재사용하는 것"** 입니다.

---

## 2. RAG가 뭔가 (왜 이 구조인가)

**RAG = Retrieval-Augmented Generation (검색 증강 생성)**

일반 챗봇은 모델이 학습한 지식으로만 답해서 ① 항공정비 같은 전문/비공개 도메인은 모르거나 ② 그럴듯하게 지어냅니다(환각).
RAG는 **답하기 전에 우리 자료를 먼저 검색**하고, **찾은 자료 안에서만** 답하게 합니다.

```
질문 → [검색] 우리 자료에서 관련 근거 찾기 → [생성] 그 근거로만 답 + 출처 표시
```

이 프로젝트는 여기에 도메인 규칙을 더했습니다: **근거가 약하면 답하지 않고 "근거 자료 없음"** 으로 처리(환각 차단).

---

## 3. 기능

| 기능 | 설명 |
|------|------|
| 의미 기반 검색 | 표현이 달라도 의미가 가까운 질문/사례를 찾음 ("압력이 낮다" ↔ "압력 상승이 안 된다") |
| 근거 기반 답변 | 검색된 자료 안에서만 답하고 **출처**(유사 세션 번호·매뉴얼 절)를 함께 표시 |
| 도메인 가드 | 항공정비 밖이거나 근거가 약하면 **"근거 자료 없음"** (무근거 추론 차단) |
| 메타데이터 필터 | 기종(KF-16/T-50/F-15K)·계통(유압/엔진/전기/통신/착륙장치)으로 검색 범위 제한 |
| LLM/폴백 자동 전환 | OpenAI 키 있으면 자연어 종합 답변, 없으면 검색 추출형 답변 (키 없이도 동작) |
| 폐쇄 루프 | 정비사가 검증한 세션을 저장 → 즉시 검색 대상에 포함 → 아카이브가 진화 |
| 채팅 UI | 말풍선형 질의응답 + 예시 질문 버튼, 검증 폼은 별도 탭으로 분리 |

## 4. 동작 흐름 (파이프라인)

```
[ 구축 단계 — 1회/데이터 변경 시 ]
 sessions.json + knowledge.json
        │  (1) 세그먼트화: 검색용 짧은 텍스트 + 표시용 전체 텍스트 분리
        ▼
   임베딩(fastembed, 다국어)  →  의미 벡터
        │  (2) 적재
        ▼
   Chroma 벡터 DB  (메타데이터: 기종/계통/증상)

[ 질의 단계 — 매 질문 ]
 정비사 질문 ──(3) 질문 임베딩──▶ Chroma 유사도 검색(Top-K) + 기종/계통 필터
        │
        ▼ (4) 도메인 가드
   유사도 ≥ 0.45 ?
     ├─ 아니오 → "근거 자료 없음"
     └─ 예 → (5) 답변 생성
                 ├─ OpenAI 키 있음 → 근거로 LLM 종합 답변
                 └─ 키 없음        → 가장 유사한 사례 추출 요약
        │
        ▼ (6) 출처 부착 (유사 세션 번호 + 매뉴얼 절)
   화면 표시

[ 폐쇄 루프 ]
 정비사 검증 → save_verified_session() → sessions.json 추가 + 인덱스 재구축 → 다시 검색됨
```

## 5. 모듈 구성 (`src/`)

| 파일 | 역할 |
|------|------|
| `config.py` | 도메인 룰셋(기종·계통·증상), 경로, 임베딩 모델명, 검색 파라미터(Top-K=5, 임계값=0.45) |
| `embedder.py` | fastembed로 텍스트 → 벡터 변환 (L2 정규화). 모델 1회 로드 캐시 |
| `index.py` | 데이터 로드 → 세그먼트화 → 임베딩 → Chroma 적재. **검색용 매칭 텍스트와 표시용 세그먼트를 분리** |
| `retriever.py` | Top-K 유사 검색 + 기종/계통 필터 + **도메인 가드**(임계값 미만 제외) |
| `generator.py` | 근거 기반 답변 생성. OpenAI(`gpt-4o`) 또는 추출형 폴백, 근거 없으면 "근거 자료 없음" |
| `pipeline.py` | 전체 오케스트레이션(`answer`) + 출처 변환 + 검증 세션 재저장(`save_verified_session`) |

`app.py` = Streamlit 채팅 UI(탭1 질의응답 / 탭2 검증·재저장).
`scripts/build_index.py` = 인덱스 구축 CLI. `scripts/deploy_hf.py` = Hugging Face 배포.

## 6. 데이터 구조

**정비 세션** (`data/sessions.json`) — 질문~조치~결과가 묶인 과거 사례:
```json
{
  "id": "H-0245", "aircraft": "KF-16", "system": "유압", "symptom": "압력 저하",
  "question": "유압 압력이 정상보다 낮게 측정됩니다. 우선 확인할 항목은?",
  "cause_candidates": ["유압 필터 막힘", "압력센서 불량", "유압 펌프 마모"],
  "action": "유압 필터 차압 확인 후 막힘 판정, 필터 교체",
  "result": "필터 교체 후 압력 정상 복귀",
  "manual_refs": ["유압 계통 점검 매뉴얼 3-2절"], "verified_by": "정비사 김민수"
}
```

**지식 자산** (`data/knowledge.json`) — 매뉴얼/FAQ/체크리스트:
```json
{ "id": "MAN-HYD-3-2", "kind": "manual", "aircraft": "KF-16", "system": "유압",
  "title": "유압 계통 점검 매뉴얼 3-2절", "content": "유압 압력 저하 시 점검 순서: ..." }
```

> 모두 **데모용 더미 데이터**입니다. 실제 정비 판단에 사용할 수 없습니다.

## 7. 핵심 설계 결정 (왜 이렇게 했나)

1. **임베딩은 로컬(fastembed, ONNX)** — API 키·`torch` 불필요. clone하면 검색이 바로 동작하고, 오프라인에서도 돕니다.
2. **모델은 `paraphrase-multilingual-mpnet-base-v2`** — 한국어 질문-질문 매칭 품질 때문. 더 작은 MiniLM은 도메인 안/밖 점수 분리가 약해 패러프레이즈를 놓쳤습니다(실측으로 확인).
3. **질문 중심 매칭 텍스트** — 임베딩은 "기종+계통+증상+질문"의 짧은 텍스트로, 표시·LLM 근거는 전체 세그먼트로 분리. 짧은 질문과 긴 사례의 매칭 희석을 막습니다.
4. **유사도 임계값 0.45** — 도메인 패러프레이즈(0.45~0.71)와 도메인 밖(0.10~0.22)을 가르는 값. 이 아래는 "근거 자료 없음".
5. **LLM은 선택, 폴백 기본** — 공개 배포엔 키를 두지 않아 비용·키 노출 위험이 없고, 로컬에선 키를 넣어 LLM 답변을 봅니다.
6. **온톨로지 대신 룰셋** — 기종·계통·증상·조치 기본 분류만 부여. 세션이 쌓이며 진화(교수님 코멘트 반영).
7. **AI는 최종 판단자가 아님** — 모든 답에 출처를 달고, 실제 조치는 정비사 검증을 전제로 합니다.

## 8. 실행 / 배포

**로컬 실행**
```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python -m scripts.build_index      # 최초 1회 (임베딩 모델 자동 다운로드)
streamlit run app.py
# (선택) OpenAI LLM 답변: cp .env.example .env 후 OPENAI_API_KEY 입력
```

**공개 배포 (Hugging Face Space, Docker SDK)**
- `Dockerfile`이 의존성 설치 + 임베딩 모델·인덱스를 **이미지 빌드 시 미리 구워** 첫 실행을 빠르게 함.
- 공개 Space에는 키를 두지 않아 **폴백 전용**으로 동작(비용 0).
- 배포: `hf auth login` 후 `python -m scripts.deploy_hf <user>/<space>`.

## 9. 한계와 확장 방향

- **한계**: 더미 데이터 기반, 실제 정비 판단 불가. 유사 사례 없으면 답하지 않음(의도적). 보안 분야 등은 범위 밖.
- **1단계(현재)**: 세션 아카이브 + RAG 질의응답.
- **2단계**: 반복 질문 분석, FAQ 자동 추천, 세그먼트 품질 개선.
- **3단계**: 데이터 축적 후 고장예측·일정 최적화·부품 수요 예측으로 확장.
